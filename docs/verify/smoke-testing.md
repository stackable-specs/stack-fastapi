# Verification: smoke-testing

Rule-by-rule conformance report for `docs/specs/practices/smoke-testing.md` adopted via [ADR-019](../adr/019-adopt-smoke-testing-as-pipeline-gate.md).

| 1   | Separate, named smoke target distinct from unit / integration / E2E                     | **Pass**                                                               | `tests/smoke/` directory; `pyproject.toml [tool.pytest.ini_options].markers` registers `smoke`; `Makefile` `smoke` target invokes `pytest tests/smoke -m smoke`. Distinct from `make test`, `make test-integration`.                                                                                                  |
| 2   | One test per business-critical path (login, primary API, health, key page)              | **Pass**                                                               | `tests/smoke/test_smoke.py` — three tests: `/health`, `/greetings/world` (primary API), `/openapi.json` (framework init). No tests for non-critical capabilities.                                                                                                                                                     |

Documented in ADR-019 Decision Outcome.                                                                                                                                            |

| 6   | Real wired system, not mocks                                                            | **Pass**                                                               | CI `smoke` job runs `docker compose up --build` (real Postgres + the just-built image) and exercises it via HTTP. `tests/smoke/conftest.py` `smoke_client` connects to `SMOKE_BASE_URL` (default `http://localhost:8000`) — no in-process app, no mocks.                                                              |

## Open follow-ups

- **Rule 8 (paging)** — wire the `smoke` job failure to the team's on-call channel (PagerDuty / Opsgenie / Slack) once the team's alerting integration is configured.
- **Rule 5 (post-deploy run)** — `.github/workflows/smoke-postdeploy.yml` is scaffolded but not yet called from a deploy workflow because no deploy workflow exists. When deployment lands, that workflow must `needs:` this one with the appropriate `base_url` and `environment` inputs.
- **Rule 12 (runbook)** — record explicit on-call escalation steps when smoke fails post-deploy, in a runbook linked from the README.

ADR-019 Status remains **Accepted** because every failing rule has a documented carve-out in the Consequences and a tracked follow-up here; no rule is silently skipped.

- File-existence checks against the stack as of this commit.

- Re-run on every change to those files or the source spec.
